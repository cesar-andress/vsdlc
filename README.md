# VSDLC Mining Pilot

Initial public research artifact for studying **AI-assisted software development provenance** on GitHub.

This repository provides a **repository mining framework** (Phases 1–2) that discovers public repositories containing AI instruction artifacts and filters a candidate **repository traceability dataset** for downstream provenance analysis.

**Research question (pilot scope):** Do repositories that publish AI instruction artifacts also retain enough release, CI, and test evidence to reconstruct traces at release-unit granularity?

**Authors:** César Andrés Sánchez, David Martin Moncunill

## What this artifact contains

| Component | Description |
|-----------|-------------|
| **Repository mining framework** | Python package `vsdlc_mining` with seed search and repository filtering |
| **Repository traceability dataset** | JSONL outputs under `data/raw/` and `data/interim/` (generated by rerunning the pipeline) |
| **Reproducibility package** | Install metadata, tests, checkpoints, and documentation for independent rerun |

Phases implemented: **Phase 1** (seed search) and **Phase 2** (repository filtering). Later stages (cloning, release-unit extraction, trace scoring) are not included in this release.

## Evidence nodes (traceability model)

The pilot targets five evidence nodes for release-unit trace reconstruction:

1. Requirement or issue ID
2. AI instruction/prompt artifact in version control
3. Code change or pull request
4. Verification artifact or CI/test evidence
5. Release tag or release decision evidence

## Installation

Requires Python **3.11+** and a GitHub personal access token.

```bash
cd ~/papers/vsdlc/vsdlc
python3.12 -m venv .venv
source .venv/bin/activate
python -m pip install -e ".[dev]"
export GITHUB_TOKEN="ghp_..."   # required — never commit this token
```

Minimal runtime install:

```bash
python -m pip install -e .
```

## Usage

### Quick pilot (controlled test run)

Use separate output paths so pilot runs do not overwrite the main corpus.

**Phase 1** — one query, two pages (~200 raw hits):

```bash
python scripts/run_seed_search.py \
  --no-resume \
  --query AGENTS.md \
  --max-pages 2 \
  --output data/raw/pilot_agents_candidates.jsonl \
  --checkpoint data/interim/pilot_agents_checkpoint.json
```

**Phase 2** — filter the pilot corpus (first 20 repos):

```bash
python scripts/run_repo_filter.py \
  --input data/raw/pilot_agents_candidates.jsonl \
  --eligible-output data/interim/pilot_agents_eligible.jsonl \
  --excluded-output data/interim/pilot_agents_excluded.jsonl \
  --summary-output data/interim/pilot_agents_summary.json \
  --limit-repos 20
```

**Expected pilot runtime:** ~2–5 minutes (dominated by GitHub code-search rate limits).

### Full run — Phase 1 (seed search)

Discovers repositories via GitHub code search, deduplicates hits, enriches metadata, and writes the primary candidate dataset:

```bash
python scripts/run_seed_search.py
```

Resume after interruption:

```bash
python scripts/run_seed_search.py --resume
```

**Expected runtime:** ~30–60+ minutes for all 17 queries plus metadata enrichment.

Checkpoint (local, gitignored): `data/interim/seed_search_checkpoint.json`

### Optional — metadata backfill before annotation

If `eligible_repos.jsonl` or `gold_sample_repos.csv` were created before description/topics/language
were persisted, run a lightweight backfill (does **not** rerun Phase 2 filtering):

```bash
export GITHUB_TOKEN="ghp_..."
python3.11 scripts/backfill_metadata.py \
  --eligible-input data/interim/eligible_repos.jsonl \
  --eligible-output data/interim/eligible_repos_enriched.jsonl \
  --gold-input data/processed/gold_sample_repos.csv \
  --gold-output data/processed/gold_sample_repos_enriched.csv
```

Use `--in-place` only when you intend to overwrite the input files (atomic writes). Recommended before independent annotation, but not required if annotators consult GitHub/README directly.

### Full run — Phase 2 (repository filter)

```bash
python scripts/run_repo_filter.py
```

Requires `data/raw/repo_candidates.jsonl` from Phase 1.

### GitHub rate limits

- **Code search** (~30 requests/minute) is the main bottleneck.
- The client throttles between pages and queries, retries with backoff, and supports `--resume`.
- **Core REST API** (`/repos/...`) has a higher hourly quota; enrichment and filtering may still slow near the limit.

## Expected outputs (repository traceability dataset)

| Path | Phase | Description |
|------|-------|-------------|
| `data/raw/repo_candidates.jsonl` | 1 | Candidate repositories with stars, license, matched paths |
| `data/interim/eligible_repos.jsonl` | 2 | Repositories passing inclusion rules |
| `data/interim/excluded_repos.jsonl` | 2 | Excluded repositories with reasons |
| `data/interim/filter_summary.json` | 2 | Aggregate filter statistics |
| `data/processed/` | — | Reserved for later pipeline stages |

## Reproducibility

See [`docs/reproducibility.md`](docs/reproducibility.md) for rerun steps, rate-limit behaviour, caching/checkpoints, and citation guidance.

- GitHub code search returns at most **1,000 results per query**.
- Outputs depend on GitHub index state at run time; exact counts may vary between reruns.

## Ethical and legal notice

- Only **public GitHub metadata** is collected through the official API.
- **No secrets** should be stored in outputs; do not commit tokens or `.env` files.
- **No repository cloning** occurs in Phases 1–2.
- Redistribution of mined metadata must respect **individual repository licenses** and the **GitHub Terms of Service**.
